# Underwriter Workbench — Application Spec

## Purpose

A canvas-first AI workbench for specialty insurance underwriters. The underwriter acts as
orchestrator, directing a set of specialist AI agents to handle legal, regulatory, research,
tooling, and inter-firm communication tasks. The app supports both individual submission
underwriting and portfolio-level management, with first-class handling of complex risk
structures (multi-layer, facultative reinsurance).

## Core Concepts

**Submission** — an incoming risk offered to the underwriter. Has one or more **Layers**
(primary, excess 1, excess 2 …). Each layer has a line size, limit, attachment point, rate.

**FacRi Panel** — when the underwriter cedes part of a layer to a reinsurer, this is modelled
as a FacRi record linked to the layer: ceded %, reinsurer name, reinsurer agent endpoint.

**Portfolio** — the aggregate of all active submissions and bound policies. Supports filtering
by line of business, cedant, territory, and exposure class.

**Agent Task** — a discrete unit of work dispatched to an agent. Has: type, input payload,
status (queued / running / complete / failed), output, and an audit trail entry.

**B2B Session** — a structured dialogue between this system's agent and an external party's
AI agent (broker, cedant, co-insurer). Uses a defined message envelope (see Agent Contracts).

**DevTool** — a containerised micro-application generated by the Developer Agent for a specific
underwriting task. Lives in its own OpenShift pod within a dedicated namespace. Can be
promoted from ephemeral (session-scoped) to persistent (saved to the underwriter's tool library).

## UI Structure

### Layout

```
┌──────────────────────────────────────────────────────────────────┐
│ Nav: [Logo] [Portfolio] [Submissions] [Settings]    [User]       │

│            │                                 │                   │

│ Chat Drawer (slides up on demand, 40vh)                          │

```

### Portfolio View (Main Canvas)

- KPI strip: total GWP, aggregate limit, largest single risk, YTD loss ratio
- Risk register table: sortable by cedant, line, territory, status, expiry
- Exposure heatmap (territory × line of business)
- Clicking a row opens Submission View

### Submission View (Main Canvas)

- **Header**: risk name, cedant, broker, line of business, status badge, action buttons
- **Risk Summary tab**: structured fields (insured, territory, coverage type, inception/expiry)
- **Layer Structure tab**: table of layers. Each row shows: layer no., limit, attachment,
  line size, premium, status. Add/edit/delete layers. FacRi sub-rows per layer.
- **Documents tab**: uploaded slips, wordings, endorsements (PDF/DOCX)
- **Agents tab**: list of dispatched Agent Tasks for this submission, with status and output
- **Audit tab**: timestamped log of all actions and agent outputs

### Context Panel

Shows a summary card for the active submission or portfolio filter. For a submission: key
stats + risk flag indicators (names clearance status, legal sign-off status, open agent tasks).

**My Tools** section (bottom of Context Panel): list of the underwriter's saved DevTools.
Each entry shows tool name, last used date, and an "Open" button that launches the tool
panel against the current submission's context.

### Agent Panel

- Lists available agents with status indicators
- "Dispatch" button per agent opens a task config drawer
- Running tasks show a live streaming output via SignalR
- Completed tasks show a summary with a "View Full Output" expand

### Chat Drawer

- Triggered via keyboard shortcut (Ctrl+Space) or footer button
- Full Claude conversation in context of the active submission or portfolio
- Can invoke agents inline via slash commands: `/legal`, `/names`, `/dev`, `/b2b`
- History persisted per submission

## Agents

### 1. Legal Agent

**Purpose**: Review policy wordings, flag non-standard clauses, check jurisdiction compliance.

**Trigger**: User uploads a wording doc, or selects a standard clause library to check against.

**Input**: `{ documentId, jurisdiction, lineOfBusiness, checklistType }`

**Output**:
```json
{
  "summary": "string",
  "flags": [{ "clause": "string", "severity": "high|medium|low", "note": "string" }],
  "recommendation": "approve | amend | escalate"
}
```

**Backend**: C# service calls Claude API with the wording text + a system prompt defining the
legal review framework. Returns structured JSON via tool_use.

---

### 2. Names Clearance Agent

**Purpose**: Check insured, broker, and cedant names against sanctions lists and regulatory
registers. Returns a clearance decision with audit evidence.

**Trigger**: User initiates check, or automatically on submission creation.

**Input**: `{ entityName, entityType, jurisdiction, submissionId }`

**Output**:
```json
{
  "clearanceStatus": "clear | refer | blocked",
  "matchedRecords": [{ "source": "string", "matchScore": 0.0, "detail": "string" }],
  "auditRef": "string"
}
```

**Backend**: Calls a configurable external sanctions API (mock in dev; real endpoint injected
via Azure Key Vault secret). Result stored in Cosmos and surfaced in the Context Panel badge.

---

### 3. Developer Agent

**Purpose**: Generate, containerise, and deploy throwaway micro-tools scoped to a specific
underwriting task (e.g. a layer pricing calculator, a cat exposure visualiser, a premium
allocation model). Tools run as real containers with controlled networking.

**Trigger**: User describes the tool they need in natural language via chat or Agent Panel.

**Input**: `{ taskDescription, contextData, networkPolicy }` — contextData is the serialised
submission; networkPolicy defines what external endpoints the tool may reach.

**Generation flow**:
1. Claude API generates the tool as a self-contained application (React + minimal Node server,
   or Python + FastAPI) based on the task description and submission context.
2. Backend wraps the generated code in a standard Dockerfile and pushes the image to the
   Azure Container Registry.
3. A C# OpenShift API client creates a new **Pod** in the `uw-devtools` OpenShift namespace,
   applying a **NetworkPolicy** that restricts egress to only explicitly permitted endpoints
   (e.g. the workbench API for context reads; no public internet by default).
4. An OpenShift **Route** is provisioned scoped to the user's session; the URL is returned
   to the frontend and rendered in an embedded panel (full iframe, not sandboxed fragment).
5. SignalR streams build and startup logs to the Agent Panel in real time.

**Tool lifecycle**:
- **Ephemeral** (default): Pod is torn down when the user closes the tool panel or the session
  ends. Image is deleted from registry after 24 hours.
- **Promoted to persistent**: User clicks "Keep this tool". The DevTool record is saved to
  Cosmos (metadata, image ref, networkPolicy, sourceCode). The pod is kept running and
  re-routed into a stable URL under the user's tool library. The tool appears in a **My Tools**
  section in the Context Panel and can be reopened against any future submission.

**Output**:
```json
{
  "devToolId": "uuid",
  "toolUrl": "https://route.openshift.../tool/{id}",
  "status": "building | running | failed",
  "imageRef": "acr.azurecr.io/devtools/{id}:latest",
  "networkPolicy": { "allowedEgress": ["string"] },
  "ephemeral": true,
  "buildLog": "string"
}
```

**Network control**: Each tool's NetworkPolicy is generated by Claude based on the task
description (e.g. a cat model tool that needs to call a specific nat-cat API endpoint gets
that endpoint whitelisted; a pure calculation tool gets egress fully blocked). The underwriter
reviews and approves the proposed NetworkPolicy before the pod is started.

**Backend**: C# `DevAgentService` orchestrates: Claude code generation → Docker build →
ACR push → OpenShift namespace provisioning → Route creation. OpenShift API credentials
stored in Azure Key Vault. All steps tracked as sub-tasks on the AgentTask record.

---

### 4. B2B Comms Agent

**Purpose**: Conduct structured AI-to-AI dialogue with an external party's agent endpoint
(broker, cedant, co-insurer) — e.g. negotiate terms on a FacRi panel, request a revised slip.

**Trigger**: User initiates a B2B session, selects counterparty, defines the mandate (what
the agent is authorised to agree without escalation).

**Message Envelope**:
```json
{
  "sessionId": "uuid",
  "from": { "firm": "string", "agentEndpoint": "url" },
  "to":   { "firm": "string", "agentEndpoint": "url" },
  "messageType": "proposal | counter | accept | reject | query",
  "payload": { },
  "timestamp": "ISO8601",
  "signature": "HMAC-SHA256"
}
```

**Flow**: Underwriter sets mandate → agent sends opening proposal → polls for response →
if within mandate, auto-accepts; if outside, surfaces to underwriter for decision → final
agreed terms written back to the submission layer.

**Backend**: C# hosted service manages the async turn-based loop. All messages stored in
Cosmos. Claude used to draft proposals and evaluate counterparty responses against mandate.

## Key Workflows

### New Submission
1. UW creates submission, fills header fields, uploads slip.
2. Names Clearance runs automatically; result shown in Context Panel.
3. UW builds layer structure (add rows, set limits/attachment/line size).
4. UW adds FacRi sub-rows if ceding.
5. UW dispatches Legal Agent on the uploaded wording.
6. Legal output reviewed; UW approves or triggers amendment.
7. UW binds the submission → status → Bound; portfolio updates.

### FacRi Negotiation via B2B
1. UW opens a layer, adds FacRi panel, selects reinsurer with a registered agent endpoint.
2. Initiates B2B session, sets mandate (max cession %, min reinsurer line size, rate range).
3. B2B agent negotiates; live transcript shown in Agent Panel.
4. Agreed terms auto-populated into FacRi record.

### Portfolio Review
1. UW opens Portfolio View, filters by territory or line.
2. Exposure heatmap highlights concentrations.
3. UW dispatches Legal Agent across a set of submissions (batch task).

### Developer Agent — Build and Keep a Tool
1. UW opens a submission, opens chat drawer, types `/dev build me a layer pricing calculator
   that takes limit, attachment, rate and shows a premium waterfall chart`.
2. Developer Agent generates the app, proposes a NetworkPolicy (egress blocked — pure calc tool).
3. UW reviews NetworkPolicy in a confirmation modal, approves.
4. Agent Panel streams build logs; tool URL appears in an embedded panel once the pod is running.
5. UW interacts with the tool, iterates via chat if needed (`/dev add a CSV export button`).
6. UW clicks "Keep this tool" → tool is promoted to persistent, named, and saved to My Tools.
7. On the next submission, UW opens My Tools → "Layer Pricing Calculator" → opens against
   the new submission's layer data automatically.

## Data Model (Cosmos DB — key documents)

- **Submission**: id, status, riskDetails{}, layers[], documents[], agentTasks[], auditLog[]
- **Layer**: id, submissionId, layerNo, limit, attachmentPoint, lineSize, premium, facriPanels[]
- **FacRiPanel**: id, layerId, cededPct, reinsurerName, reinsurerAgentEndpoint, agreedTerms{}
- **AgentTask**: id, submissionId, agentType, status, input{}, output{}, subTasks[], createdAt, completedAt
- **DevTool**: id, ownerId, name, description, imageRef, networkPolicy{}, sourceCode, ephemeral,
  podName, routeUrl, lastUsedAt, submissionContextSchema{}
- **B2BSession**: id, submissionId, counterparty{}, mandate{}, messages[], status, finalTerms{}
- **PortfolioSnapshot**: id, generatedAt, kpis{}, exposureMatrix[][]

## Out of Scope (v1)

- Claims handling or FNOL
- Full policy lifecycle (endorsements, renewals)
- Real sanctions API integration (mock only in v1)
- Live external reinsurer agent endpoints (simulated responder in dev)
- Role-based access beyond single underwriter persona
- Mobile layout
- DevTool versioning or collaborative sharing of saved tools (post-v1)
- Auto-scaling of DevTool pods under load (single replica in v1)
