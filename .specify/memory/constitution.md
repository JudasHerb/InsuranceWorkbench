# Underwriter Workbench Constitution

## Core Principles

### I. Canvas-First UX

The underwriter is the orchestrator. The main canvas is the primary interaction surface for all
underwriting work. All meaningful actions (layer editing, agent dispatch, document review) MUST
occur within the canvas context — not in standalone modals or separate pages that lose submission
context. Navigation MUST remain flat: Portfolio View → Submission View. The chat drawer and
agent panel are always reachable without navigating away from the active submission.

The "My Tools" section in the Context Panel is the entry point for saved DevTools. Launching a
DevTool MUST open it in an embedded panel within the current canvas — not as a separate tab or
window that breaks submission context.

**Rationale**: Underwriters manage complex, multi-layered risks under time pressure. Context
switching is a cognitive tax. Every navigation step that loses the current submission context
reduces the tool's value.

### II. Agent-Mediated Automation

All automated work MUST be dispatched as a typed `AgentTask` document. Agents MUST be either
explicitly dispatched by the underwriter or triggered by a defined system rule (e.g., names
clearance on submission creation). Silent background operations with no audit trace are
prohibited. Every agent invocation MUST produce a persisted `AgentTask` record with:
`agentType`, `status`, `input`, `output`, `subTasks[]`, `createdAt`, and `completedAt`.

Multi-step agent flows (e.g., Developer Agent: generate → build → push → deploy) MUST track
each step as a sub-task entry on the parent `AgentTask` record.

Inline agent invocation via slash commands (`/legal`, `/names`, `/dev`, `/b2b`) in the chat
drawer is permitted but MUST still produce an `AgentTask` record.

**Rationale**: Regulatory environments require a demonstrable decision trail. An agent that
acted without a record is indistinguishable from an agent that never ran.

### III. Compliance and Audit by Design

Names clearance MUST run automatically on every new submission before the underwriter can
proceed to layer building. Legal review output MUST be visible in the Audit tab before a
submission can be bound. Every state transition on a `Submission` (created → in-review → bound)
MUST append an `auditLog` entry with a timestamp, actor (user or agent), and summary.

All agent outputs that inform an underwriting decision (legal flags, clearance decisions,
negotiated FacRi terms) MUST be stored in Cosmos DB and surfaced with an `auditRef`. Ephemeral
DevTools (not promoted to persistent) are exempt from source-code persistence but their
`AgentTask` record (including the proposed NetworkPolicy and approval event) MUST be retained.

**Rationale**: Specialty insurance is a regulated industry. Audit trails are not optional
features — they are compliance requirements.

### IV. Security and Sandboxing

The following security rules are NON-NEGOTIABLE:

**Developer Agent (container isolation model)**:
- Generated tool code MUST run inside a dedicated OpenShift Pod in the `uw-devtools` namespace.
  Code MUST NOT be executed on the backend application server or in a browser iframe.
- Every pod MUST have a **NetworkPolicy** generated by the Developer Agent that restricts egress
  to only the endpoints required for that tool's stated purpose. Default stance is egress fully
  blocked; additional endpoints require explicit whitelisting.
- The underwriter MUST review and explicitly approve the proposed NetworkPolicy in a confirmation
  step before the pod is started. The approval event MUST be logged on the `AgentTask` record.
- OpenShift API credentials and Azure Container Registry credentials MUST be stored in Azure Key
  Vault and injected at runtime. No credentials in source code or environment files committed to
  the repository.

**B2B Communications**:
- B2B session messages MUST be signed with HMAC-SHA256 using a shared secret. Unsigned or
  invalid-signature messages MUST be rejected without processing.
- HMAC secrets MUST be stored in Azure Key Vault.

**Authentication**:
- Authentication MUST use Azure Entra ID (MSAL). No custom auth flows.
- All other secrets (Claude API keys, sanctions API credentials) MUST be stored in Azure Key
  Vault and injected at runtime.

**Rationale**: The workbench handles sensitive risk data and executes AI-generated code that can
initiate network connections and make financial commitments. Container isolation with explicit
NetworkPolicy approval provides stronger and more auditable sandboxing than browser-level iframe
restrictions, while maintaining underwriter visibility and control over what each tool can reach.

### V. Bounded Scope (v1)

The following are OUT OF SCOPE for v1 and MUST NOT be built, even if technically
straightforward:

- Claims handling or FNOL
- Full policy lifecycle (endorsements, renewals)
- Real sanctions API integration (mock provider only)
- Live external reinsurer agent endpoints (simulated responder in dev/test only)
- Role-based access beyond single underwriter persona
- Mobile layout
- DevTool versioning or collaborative sharing of saved tools between underwriters
- Auto-scaling of DevTool pods under load (single replica per tool in v1)

New features MUST be evaluated against this boundary. Any deviation requires an explicit
constitution amendment before implementation begins.

**Rationale**: Scope creep in a domain-heavy application produces a system that does many
things poorly. v1 delivers a focused, demonstrable tool; expansion follows validated delivery.

### VI. Test Discipline

- Playwright MUST be used for all E2E tests covering key user journeys (submission creation,
  agent dispatch, binding flow, DevTool build-and-keep workflow).
- xUnit MUST be used for all backend unit and integration tests.
- Agent integrations MUST have contract tests that run against mock Claude API responses —
  not against the live API — to ensure deterministic CI.
- The Names Clearance agent MUST use a mock sanctions API in all non-production environments.
  Tests MUST NOT call external sanctions endpoints.
- The Developer Agent's OpenShift API calls (pod creation, route provisioning, NetworkPolicy
  application) MUST be mocked in CI. Tests MUST NOT create real pods or push real images.
- Azure Container Registry push MUST be mocked in CI using a fake registry stub.
  The image build step MAY run locally in Docker but MUST be skippable in CI.
- The NetworkPolicy approval flow (underwriter review → approve → pod start) MUST have
  an integration test covering the approval-gating behaviour.

**Rationale**: AI agent outputs are non-deterministic and container infrastructure is
expensive and slow to provision. Mocking external surfaces (Claude API, OpenShift, ACR)
keeps CI fast, deterministic, and free of infrastructure dependencies.

## Technical Stack Constraints

The following technology choices are fixed for v1. Proposals to replace a tier require a
constitution amendment.

| Layer          | Technology                                                        |
|----------------|-------------------------------------------------------------------|
| Frontend       | React 18 (TypeScript), Vite, TailwindCSS                         |
| Backend        | ASP.NET Core 9, C#, REST + SignalR (streaming)                    |
| Database       | Azure Cosmos DB (NoSQL, document model)                           |
| AI             | Anthropic Claude API (`claude-opus-4-6` for agent tasks)          |
| Testing        | Playwright (E2E), xUnit (backend unit/integration)                |
| Hosting        | Azure Container Apps (main application)                           |
| Orchestration  | OpenShift — `uw-devtools` namespace for Developer Agent tool pods |
| Container Reg. | Azure Container Registry (DevTool images)                         |
| Auth           | Azure Entra ID (MSAL)                                             |
| Secrets        | Azure Key Vault (all credentials injected at runtime)             |

**Streaming**: Real-time agent output and DevTool build logs MUST use SignalR. Polling is not
an acceptable substitute for live task status in the Agent Panel.

**Document storage**: Uploaded slips, wordings, and endorsements (PDF/DOCX) are stored as
references within the `Submission` document in Cosmos DB. Blob storage strategy is a v1
implementation detail and MUST be decided in the feature plan — not deferred to runtime.

**DevTool persistence**: Promoted (persistent) DevTools are stored as `DevTool` documents in
Cosmos DB containing: `id`, `ownerId`, `name`, `imageRef`, `networkPolicy{}`, `sourceCode`,
`podName`, `routeUrl`, `lastUsedAt`, and `submissionContextSchema{}`.

## Development Workflow and Quality Gates

### Feature Development Flow

1. Feature MUST have a spec (`spec.md`) before planning begins.
2. Plan (`plan.md`) MUST include a Constitution Check section that verifies compliance with all
   six principles before implementation tasks are written.
3. Tasks (`tasks.md`) MUST be organized by user story so each story is independently
   implementable, testable, and demonstrable.
4. Agent-related tasks MUST include a contract test task (against mock responses) before the
   implementation task.
5. Developer Agent tasks MUST include a task for the NetworkPolicy generation and approval flow
   before the pod-provisioning task.

### Quality Gates Before Merge

- All Playwright E2E tests pass.
- All xUnit tests pass.
- Agent contract tests pass against mocked Claude responses.
- Developer Agent OpenShift and ACR calls verified against mocks (no real pods created in CI).
- NetworkPolicy approval-gating integration test passes.
- No secrets detected in committed files (enforced by pre-commit hook or CI scan).
- Constitution Check in `plan.md` is completed and shows no unresolved violations.

### Complexity Justification

Any implementation that introduces a pattern not derivable from the spec (e.g., a new
abstraction layer, a caching tier, a background job) MUST be justified in the plan's
Complexity Tracking table with: the violation, the reason it is needed, and the simpler
alternative that was rejected.

## Governance

This constitution supersedes all other development practices and conventions for the
Underwriter Workbench project. In conflicts between this document and any other guide,
this constitution takes precedence.

**Amendment procedure**: Amendments MUST be proposed as a diff to this file, accompanied by:
(a) the principle or section being changed, (b) the motivation, and (c) a migration note for
any in-flight features affected. The amendment MUST increment the version according to semantic
versioning rules (see below) and update `LAST_AMENDED_DATE`.

**Versioning policy**:
- MAJOR: Principle removal, redefinition, or backward-incompatible scope change.
- MINOR: New principle or section added, or material expansion of existing guidance.
- PATCH: Clarification, wording fix, non-semantic refinement.

**Compliance review**: Every `plan.md` MUST include a Constitution Check section. PRs that
introduce agent types, external integrations, container workloads, or new data persistence
patterns MUST be reviewed against Principles II, III, and IV before merge approval.
